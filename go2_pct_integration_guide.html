xxx 与 PCT_planner 的联动架构说明

这套系统不是一个单仓库完成全部导航的结构，而是一个 分层协作 的三段式架构： xxx 负责在线建图与局部避障执行， PCT_planner 负责基于 tomogram 的全局规划， 机器人本体提供实时点云、里程计与最终运动执行接口。

一句话总结
xxx 先把 Go2 现场环境积累成 .pcd 地图， PCT_planner 再把这张地图转成 tomogram 做跨楼层全局路径， 最后 xxx 订阅 /pct_path，结合实时点云做 EGO 局部避障与速度执行。

Critical Assumption map 与 odom 需要数值对齐，Go2Map 模式下由静态 TF 保证
Workspace A

xxx 的第一重身份: 建图工作区

  • 订阅 /utlidar/cloud_deskewed 与 /utlidar/robot_odom。
  • 在线累积点云，运行结束后输出 output/go2_built_map.pcd。
  • 可选记录命名点位，生成 output/go2_named_poses.json。
  • 这张 PCD 不是局部避障直接使用的地图，而是给 PCT_planner 做全局理解的离线输入。
Workspace B

PCT_planner 的身份: 全局规划工作区

  • 读取 .pcd 并生成 tomogram 缓存 .pickle。
  • Go2Map 模式下，用 /utlidar/robot_odom 作为规划起点。
  • 通过 RViz 的 /clicked_point、/move_base_simple/goal、/pct_3d_goal_pose 接收目标。
  • 把全局路径发布为 /pct_path，路径坐标系是 map。
Workspace A Again

xxx 的第二重身份: 局部规划与执行工作区

  • 订阅 /pct_path 作为上游参考路径。
  • 同时持续读取实时点云和里程计，维护局部占据图。
  • 使用 EGO-Planner 做局部重规划、碰撞检测与紧急停止。
  • 输出 planning/bspline、/cmd_vel，最终转成 /api/sport/request。

文件级联动

这部分发生在“运行前”或“环境更新后”。它决定 PCT_planner 看见的世界是什么样。

1. xxx 输出现场地图
/home/huang/xxx/output/go2_built_map.pcd
2. PCT_planner Go2Map 启动脚本接收该文件
./scripts/run_ros2_demo.sh Go2Map /home/huang/xxx/output/go2_built_map.pcd
3. 脚本同步到 PCT 的缓存目录
/home/huang/PCT_planner/rsc/pcd/go2_built_map.pcd
4. tomography 生成 tomogram
/home/huang/PCT_planner/rsc/tomogram/go2_built_map.pickle
这意味着 PCT_planner 并不直接消费 xxx 在线发布的点云地图话题，它消费的是 xxx 产出的 PCD 文件及其 tomogram 缓存。

话题级联动

这部分发生在“运行时”。它决定机器人此刻往哪里走、局部如何躲障。

  • PCT_planner 在 /pct_path 上发布 nav_msgs/Path。
  • xxx/ego_planner 订阅该路径，内部话题名是 reference_path，通过 launch remap 到 /pct_path。
  • 本地 EGO 同时订阅 /utlidar/cloud_deskewed 与 /utlidar/robot_odom 做局部占据图和安全检查。
  • go2_traj_server 根据局部 B-spline 生成 /cmd_vel。
  • unitree_sport_bridge 把 /cmd_vel 转成 /api/sport/request 发给机器狗。
一个常见误解
/pct_path 不是最终轨迹，也不会直接发给机器人执行。它只是“全局参考线”。真正上狗的是 xxx 本地重新优化后的局部轨迹和速度命令。

坐标系与 TF 约定

对象 坐标系 来源 为什么重要
PCT 发布的 /pct_path map traj2ros(..., frame_id="map") 全局路径名义上在 map 坐标系下
PCT RViz 固定坐标系 world PCT 一键脚本启动 world -> map 静态 TF 主要是为了 RViz 显示稳定
Go2Map 模式下补充 TF map -> odom PCT 一键脚本启动 identity 静态 TF 为 xxx 局部规划消费路径创造数值对齐条件
xxx 局部规划局部地图 odom go2_navigation.yaml 中 grid_map/frame_id: odom 局部避障和里程计一致
这里有一个非常关键的“代码现实”
xxx 中的 referencePathCallback 读取 /pct_path 时，直接拿路径点的数值做规划， 并没有额外做 TF 变换。也就是说，当前联动默认依赖 map 与 odom 数值上基本重合。 这也是 PCT 的 Go2Map 脚本要起 map -> odom identity 静态 TF 的根本原因。

运行时完整时序

  1. Go2 运行时持续提供 /utlidar/cloud_deskewed 和 /utlidar/robot_odom。
  2. xxx 的 go2_map_builder 把现场扫描积累为 go2_built_map.pcd。
  3. PCT_planner 读取这张 PCD，生成 Go2Map 的 tomogram；如果缓存存在且没过期，就直接重放。
  4. PCT_planner 在 Go2Map 模式下，把 /utlidar/robot_odom 当作全局规划起点。
  5. 你在 RViz 里点选 3D 目标点或 2D 目标点，PCT_planner 立即重规划并发布 /pct_path。
  6. xxx 的 EGO 本地规划器接收到 /pct_path，从全局路径上提取局部目标段。
  7. xxx 再结合实时点云构建局部占据图，对全局路径做局部重规划、绕障、必要时急停。
  8. 局部规划结果先发布成 planning/bspline，再由 go2_traj_server 变成 /cmd_vel。
  9. unitree_sport_bridge 把 /cmd_vel 转成 /api/sport/request，机器人执行。
职责边界非常清晰
PCT_planner 负责“从地图和目标中得出一条能到达的全局路”。
xxx 负责“在真实现场里，把这条全局路变成安全、可执行、会绕近处障碍的局部运动”。

为什么一定要分成这两个工作区

  • 建图与全局规划的时间尺度不同。 xxx 输出的是现场逐步积累出来的点云地图；PCT_planner 使用它做 tomogram 级的结构理解。
  • 全局规划与局部避障的输入不同。 PCT_planner 主要看静态场景结构；xxx 本地避障主要看实时点云和 odom。
  • 全局路径与运动控制之间需要一个本地执行层。 只靠 /pct_path 并不能让 Go2 自动躲临时障碍，也不能直接变成可走的速度命令。
  • 这个架构天然支持跨楼层 + 近场避障。 PCT 解决楼层与大尺度拓扑，EGO 解决近场安全。

这套架构的实际优点

  • 全局规划可以利用 tomogram 做多层结构理解，不必在局部规划里硬扛跨楼层问题。
  • 局部规划继续吃实时点云，所以前方突然出现人、椅子、小车时，仍能本地绕开或急停。
  • 建图、全局规划、局部执行三者解耦，调试时可以单独定位问题出在哪一层。
  • 你可以替换上游全局规划器，只要继续发布 /pct_path，xxx 本地执行层仍可复用。

推荐启动方式

阶段 A: 用 xxx 采集并保存地图

cd ~/xxx
source /opt/ros/jazzy/setup.bash
source ~/xxx/install/setup.bash

ros2 launch go2_mapping_minimal mapping_go2_builtin.launch.py

这一步的目标是拿到 /home/huang/xxx/output/go2_built_map.pcd。如果只是为了生成地图文件，不必把它理解成最终的 PCT 联动运行模式。

阶段 B: 用 PCT_planner 跑 Go2Map 全局规划

cd ~/PCT_planner
source /opt/ros/jazzy/setup.bash
source ~/PCT_planner/install/setup.bash

./scripts/run_ros2_demo.sh Go2Map \
  /home/huang/xxx/output/go2_built_map.pcd

这一步会同步 PCD、生成或复用 tomogram、起静态 TF、打开 RViz、并在目标设置后发布 /pct_path。

阶段 C: 用 xxx 跑真正的 PCT -> EGO 局部执行

cd ~/xxx
source /opt/ros/jazzy/setup.bash
source ~/unitree_ros2/cyclonedds_ws/install/setup.bash
source ~/xxx/install/setup.bash

ros2 launch go2_navigation simple_goal_nav.launch.py

这是 路径模式。它会订阅 /pct_path 做局部执行，才是和 PCT_planner 联动时更匹配的入口。

人工旁路模式

ros2 launch go2_navigation click_goal_nav.launch.py

# 或
ros2 launch go2_mapping_minimal mapping_go2_builtin.launch.py

这两个 launch 当前都把 ego_planner_node 强制设成 fsm/flight_type = 1，也就是直接接收 /goal_pose 的手工目标模式，更适合单仓库调试，而不是主 PCT 联动入口。

当前代码现状与容易踩坑的点

现象 真实原因 建议
以为 mapping_go2_builtin.launch.py 会直接吃 /pct_path 这个 launch 虽然 remap 了 reference_path，但又覆盖了 fsm/flight_type = 1，实际走的是手工 goal 模式 联动 PCT 时优先使用 simple_goal_nav.launch.py，或者显式把 flight type 调回路径模式
PCT 发布了 /pct_path，但 xxx 走偏或不对齐 路径在 map，本地规划以 odom 为参考，当前实现默认依赖二者数值一致 保持 Go2Map 的 map -> odom identity 约定，或后续给 xxx 增加显式 TF 变换
以为保存的 PCD 会直接参与局部避障 xxx 的局部避障看的是实时点云，不是离线 PCD 把 PCD 理解为全局规划输入，把实时点云理解为局部安全输入
同时跑多个本地规划 launch，机器人行为混乱 多个节点可能同时竞争 /cmd_vel 或 /api/sport/request 联动时保持“一个全局规划器 + 一个本地执行器 + 一个 bridge”即可
最应该记住的一条
PCT 联动模式的主入口不是 click_goal_nav.launch.py，而是 simple_goal_nav.launch.py。 前者是“直接手工目标点驱动局部规划”，后者才是“上游给路径，本地做执行”的模式。

最后的架构结论

  • xxx 不是纯建图仓库。 它既负责在线建图，又负责真正把全局路径变成 Go2 的可执行局部运动。
  • PCT_planner 不是纯可视化仓库。 它承担跨楼层、结构化、长程全局规划的核心职责。
  • 这两个仓库的边界是非常自然的。 一个更贴近现场实时执行，一个更擅长基于环境结构做长距离规划。
  • 真正的联动主线是： PCD file -> tomogram -> /pct_path -> local B-spline -> /cmd_vel -> /api/sport/request。
本页基于当前两个工作区的实际代码组织和 launch 行为整理而成。 其中一个特别重要的现实约束是：/pct_path 的 frame 是 map， 而 xxx 的本地规划内部按 odom 数值直接取点，因此目前的联动默认依赖 Go2Map 模式下的 map -> odom identity 约定。
